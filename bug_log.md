## 2026-07: project_peak_demand: Duplicate `<head>`/`<body>` breaking the D3 counter

**Project:** project_peak_demand
**File:** index.html
**Symptom:** The D3 animated number counter ("What 1 GW Means" section) didn't load reliably: inconsistent behavior on page load.

### What was wrong

Per my TA's diagnosis: removed the duplicate `</head>` and `<body>` tags from the middle of the page. Should only use them once.

I had pasted a chunk of code (originally built for CodePen) directly into the middle of my `index.html`. That chunk included its own `</head>` and `<body>` tags, which meant my page ended up with **two sets** of these tags: one at the top of the file (correct), and a second set buried in the middle, in the "What 1 GW Means" section.

### The fix

1. Removed the extra `</head>` and `<body>` tags from the middle of the page, so each tag appears exactly once, in its correct place.
2. Added an `IntersectionObserver` so the counter animation only starts once the "What 1 GW Means" section is 25% visible on screen, instead of firing immediately on page load (which meant it could finish counting before I'd even scrolled down to see it).
3. **Important detail from my TA:** the `IntersectionObserver` only works correctly if it's kept *inside* the D3 animation function: not set up separately outside of it.

Comments marked with my name are in the code where the new parts were added, for reference.

### Takeaways

- **HTML documents can only have ONE `<head>` and ONE `<body>`, period.** I understood this in theory, but didn't actually notice I'd broken it, because the browser didn't throw a loud error: it just silently misbehaved. That's the part I want to remember: broken HTML structure doesn't always announce itself with an error message the way broken code does. (!!!)

- **Copy-pasting code from one context (CodePen, a full standalone file) into another (a section of an existing page) isn't a safe copy-paste.** A full HTML file and a fragment meant to live *inside* an existing file are not the same shape, even if the visible content looks identical. I need to check what I'm actually pasting, not just what it renders as.

- **Where code lives inside a function matters, not just whether it "runs."** The `IntersectionObserver` had to be nested *inside* the D3 animation logic to work: putting it outside broke the connection between "the section became visible" and "start the animation." I don't think I fully understand *why* yet: noting this to come back to and actually learn, not just fix. (Again I get it conceptually but in practice it was hard to fix) (This is perhpas the most important lesson!!!)

Additional notes:

Why only one of each: an HTML page has exactly one root element, <html>, and it's built to have exactly two sections inside it — <head> (information about the page: title, styles, which scripts to load) and <body> (the actual visible content). That's not a rule someone arbitrarily enforces — it's baked into what the tags mean. There's no such thing as "the page's second beginning" or "the second set of settings for this page," the same way a book doesn't have two title pages. One page, one set of settings, one body of content.
What actually happens if you duplicate them — and this is the important part: browsers are extremely forgiving. Unlike, say, Python (which stops cold and shows you an error if your code is malformed), a web browser's job is to render something no matter how broken the HTML is — because the web is full of imperfect pages, and a browser that refused to show anything on a small mistake would be useless. So when a browser's parser hits a second <body> tag, it doesn't create a second real body — it quietly folds everything that follows into the one real <body> that already exists, and just ignores the redundant tag itself.
This is exactly why your bug was silent instead of throwing an error. Your page never actually had two bodies — the browser merged everything into one. But your two <style> blocks (one from the top of your file, one from the pasted-in chunk) both still got read and applied to that single merged body. Since both had a rule targeting plain body { ... }, and CSS's rule is "whichever rule appears later in the file wins when there's a tie," your second block's margin: 0 silently overrode the first block's margin: 40px auto — no crash, no warning, just your centering quietly disappearing.
The one-sentence version for your own notes: duplicating structural tags like <head>/<body> doesn't create two of something — it creates one broken something, merged in ways you didn't intend, with the browser never telling you it did that.
